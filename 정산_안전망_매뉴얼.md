## 1️⃣ 1단계 — 시점 자동 스냅샷

**무엇을 하나**
- 정산 데이터가 변경될 때마다 자동으로 시점 백업
- 30분마다 주기적으로도 백업
- 60일 보관 (24h 전부 / 7d 일 1개 / 60d 주 1개)

**테스트 절차**
1. `Ctrl + Shift + R` 강제 새로고침
2. 정산 탭 진입 → F12 콘솔에 **`[정산 안전망] 30분 주기 자동 스냅샷 가동`** 메시지 확인
3. 정산 액션바의 **`[🕐 시점 복구]`** 버튼 클릭 → 모달이 열려야 함
4. 모달 안에서 **`[+ 지금 백업]`** 클릭 → 좌측 목록에 새 스냅샷 생성됨
5. 스냅샷 클릭 → 우측 미리보기에 데이터 차이 비교 표시
6. **`[⚠ 이 시점으로 복구]`** 누르면 그 시점으로 데이터 복원 (직전 상태는 자동 백업되므로 되돌리기 가능)

---

## 2️⃣ 2단계 — 유실 감지 경보

**무엇을 하나**
- 정산 탭 진입 시 24시간 이내 안정 스냅샷과 자동 비교
- `-10% 이상` 또는 `-20건 이상` 감소 시 빨강 모달 즉시 노출

**테스트 절차**
1. 정산 탭 진입 → 30분 기다리거나, `[+ 지금 백업]` 한 번 눌러 스냅샷 생성
2. 정산 데이터를 20건 이상 의도적으로 삭제 (또는 큰 월 필터 좁히기로 의도적 감소 만들기)
3. 다른 탭(예: 배차 등록) 갔다가 다시 정산 탭으로 진입
4. **빨강 경보 모달**이 떠야 정상
   - 카드 3개: 직전 안정 시점 / 현재 / 감소량
5. 모달의 `[🕐 시점 복구 열기]` 누르면 시점 복구 모달로 즉시 전환
6. `[이번 세션은 무시]` 누르면 같은 세션 동안 다시 안 뜸 (새로고침 시 재경고)

**오탐 방지 (경보 OFF 조건)**
- 첫 사용자 / 스냅샷 없음
- 현재 데이터 0건 (로딩 전)
- 이번 세션에 이미 무시함

## 3️⃣ 3단계 — 시트 동시 쓰기 Lock

**무엇을 막나**
- 여러 PC가 동시에 정산 시트에 쓸 때 마지막 쓰기만 살아남고 앞 쓰기가 사라지는 race condition
- 12월 주류 누락 사고의 유력한 원인 중 하나

**클라이언트 측 (ERP) — 이미 적용**
- 시트 push 응답이 `locked:true`면 자동 재시도 (최대 2회, 3초 간격)
- 토스트로 진행 상황 안내: `🔒 다른 PC 작업 중 — 자동 재시도 1/2`
- 끝까지 실패해도 데이터 손실 없음 (다음 변경 시 또 시도됨)

**Apps Script 측 — 보스가 직접 적용 필요**
- 가이드 파일: `APPS_SCRIPT_정산LOCK.md`
- 핵심: `doPost(e)` 맨 위에 `LockService.getScriptLock()` 블록 추가
- 정산 관련 action만 보호 (`saveSettlement`, `saveSettlementBackup` 등)
- 적용 안 해도 ERP는 정상 작동 (Lock 없이 그대로 호출됨)

**테스트 절차 (Apps Script 적용 후)**
1. PC A에서 `[💾 시트 백업]` 누름
2. PC A가 진행 중일 때 PC B에서도 같은 버튼 누름
3. PC B에서 토스트 `🔒 다른 PC 작업 중 — 자동 재시도 1/2` 떠야 함
4. PC A 완료 후 PC B 자동 진행 → 둘 다 데이터 시트에 반영됨

## 4️⃣ 4단계 — 변경 로그 누적 기록

**무엇을 추적하나**
- 정산 데이터가 변경될 때마다 직전 스냅샷과 자동 diff 비교
- 누가 · 언제 · 몇 건 추가/삭제/수정했는지 60일 추적
- 영향받은 코드(date|dispCode)도 최대 10개씩 샘플 기록

**테스트 절차**
1. 정산 탭 액션바의 보라색 **`[📜 변경 이력]`** 버튼 클릭
2. 모달이 열리고 변경 이력 목록 표시
3. 필터 변경: 전체 / 변경 있음만 / 큰 감소만 (-10건↓)
4. 각 row에 작업자 + 추가/삭제/수정 카운트 + 영향 코드 칩 표시 확인

**12월 사고 같은 거 또 나면 (대응 절차)**
1. 정산 탭 → `[📜 변경 이력]` 열기
2. 필터: **`큰 감소만`** 선택
3. **누가 · 언제 · 어떤 코드를 삭제했는지** 즉시 확인 가능
4. `[🕐 시점 복구]`로 직전 안정 상태 복구
5. 사고 원인 분석: race condition인지 / 직원 실수인지 / 시트 timeout인지 판단

## 🔧 트러블슈팅

### "시점 복구 모달이 열리지 않음"
- `Ctrl + Shift + R` 강제 새로고침
- F12 콘솔 에러 메시지 확인
- IndexedDB 권한 확인 (브라우저 시크릿 모드면 작동 안 함)

### "변경 이력에 아무것도 안 보임"
- 스냅샷이 2개 이상 있어야 diff 비교 가능 (첫 스냅샷만 있으면 비교 대상 없음)
- `[+ 지금 백업]`을 한 번 더 누르고 → 정산 데이터 변경 → 다시 변경 이력 확인

### "빨강 경보 모달이 안 뜸 (감소했는데도)"
- 24시간 이상 된 스냅샷만 있으면 비교 안 함
- 이번 세션에 이미 무시함 → 새로고침
- F12 콘솔에서 `checkSettlementDataLoss()` 직접 호출해서 강제 점검 가능

### "시트 백업 시 계속 Lock 토스트"
- Apps Script가 LockService 적용된 후, 다른 PC가 30초 이상 무거운 작업 중일 가능성
- 30초 후 수동 재시도
- 또는 PC들끼리 "지금 백업 누른다" 한마디 맞춰서 충돌 방지
